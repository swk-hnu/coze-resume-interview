# 项目说明：工作流流程原理、前端搭建与问题解决

适用于组内汇报 / PPT 内容整理

## 一、项目整体目标

本项目是一个“Coze 工作流 + 前端交互界面”的组合应用，主要面向两个真实求职场景：简历优化 / 生成，以及模拟面试。前端负责收集用户输入、上传简历、控制交互流程和展示结果；Coze 工作流负责真正的 AI 处理、提问、评分、诊断和报告生成。

- 核心任务 1：简历优化 / 生成
- 核心任务 2：模拟面试
- 整体特点：文件上传、工作流调用、多轮会话、结果格式化展示

## 二、工作流的流程原理

可以把工作流理解成整个系统的“智能处理引擎”。前端不直接生成简历结论或面试题，而是负责把用户输入整理成参数，再交给 Coze 工作流执行。

### 1. 简历优化流程

1. 用户选择“生成 / 优化简历”，输入目标岗位并上传简历文件。
2. 前端识别当前任务类型为 diagnosis，并提取目标岗位、补充说明、简历文件等信息。
3. 前端会先尝试提取简历正文内容，再把文件对象和正文文本一起传给工作流。
4. Coze 工作流根据目标岗位和简历内容生成诊断结果。
5. 前端把返回内容格式化展示为综合评分、结构完整性分析、内容质量分析、岗位匹配度分析和优先改进建议。

### 2. 模拟面试流程

1. 用户选择“模拟面试”，输入目标岗位并上传简历文件。
2. 前端先要求用户输入面试轮次，例如 2 轮或 3 轮。
3. 前端启动 Coze 工作流，工作流返回第 1 题。
4. 用户回答后，前端再次调用工作流提交回答，获取本轮反馈、评分和下一题。
5. 如果还有剩余轮次，前端继续追问；如果已经结束，则展示模拟面试复盘报告。

## 三、关键参数设计（方便理解工作流）

为了让工作流稳定识别不同场景，前端在调用时设计了一组较完整的参数。核心思路是：一套参数同时覆盖“任务意图、岗位信息、会话信息、简历信息、历史问答信息”。

### 1. 基础业务参数

- user_message：用户补充说明
- target_position：目标岗位，例如“产品经理”
- intent：任务类型，取值一般为 diagnosis 或 interview
- session_id：本次会话的唯一标识，用来串联同一轮任务
- total_rounds：模拟面试总轮次
- current_round：当前是第几轮
- question：当前问题
- answer：用户当前回答
- history：历史问答记录
- user_answers：历史回答数组

### 2. 简历相关参数

这是整个项目里最关键的一部分。因为单纯上传文件并不能保证工作流稳定读到简历内容，所以系统会同时传递文件对象和文本内容。

- a / resume_file / resume / file：上传后的文件对象，本质上依赖 file_id
- document：文件内容的兼容字段
- resume_text：前端提取出的简历正文
- resume_content / content：和 resume_text 对应的兼容字段
- resume_name：简历文件名

这样做的目的，是提高工作流节点对简历内容的识别成功率，避免只传一个字段时出现“工作流调用成功，但没有读到简历”的问题。

### 3. 组合消息参数

前端还会把用户诉求、目标岗位、补充说明、轮次、当前问题、当前回答等信息拼接成一段完整消息，统一传给工作流，帮助模型更稳定地理解上下文。

## 四、前端是如何搭建的

本项目的前端采用原生 HTML + CSS + JavaScript 搭建，优点是轻量、易理解、易部署。前端并不是一个单页营销页面，而是一个面向实际使用场景的交互工具。

### 1. 页面结构

- 首页表单区：用于选择任务类型、填写目标岗位、上传简历、补充说明。
- 聊天交互区：用于展示简历优化结果，或继续进行模拟面试的多轮问答。
- 配置区：开发阶段可手动填写 Coze API 地址、工作流 ID、Token，并读取工作流参数说明。

### 2. 前端状态管理

前端通过统一状态对象记录当前流程，避免每一步都从头开始。状态中包含 phase、sessionId、requestType、targetPosition、rounds、currentRound、resumeFile、interviewHistory、lastQuestion、busy 等字段。

这样做的意义是：前端可以知道当前流程进行到哪里、已经上传了什么、还差什么、下一步应该向工作流发送什么参数。

### 3. 文件处理逻辑

- 支持上传 PDF / DOC / DOCX / TXT。
- TXT 文件可直接读取纯文本。
- DOCX 文件通过 mammoth 提取正文内容。
- 提取出的正文会先进行清洗，再截取合理长度传给工作流。

### 4. 结果展示逻辑

前端不会把工作流返回的 JSON 原样直接展示，而是根据内容类型整理成更适合阅读的结果，例如：简历诊断报告、当前轮反馈、下一题、模拟面试复盘报告等。这样可以显著提升用户体验。

## 五、遇到的主要问题与解决方法

### 问题 1：简历优化返回 0 分，像是没有读到简历

现象：工作流被成功调用，但返回结果常常是“未识别到有效简历内容”“结构缺失”“内容为空”。

原因：一开始主要依赖上传文件后的 file_id，工作流并不一定能稳定解析 doc/docx 的正文内容。

解决方法：前端增加本地提取简历正文的能力，把解析出来的内容同时作为 resume_text、resume_content、content 等字段传给工作流。这样即使文件解析不稳定，工作流仍能直接读取简历文本。

### 问题 2：模拟面试只能进行一轮，不能继续出第二题

现象：第 1 题能生成，回答后也能评分，但流程直接结束。

原因：工作流在不同阶段的返回格式不完全一致。有时回答后会直接给下一题，有时只给本轮反馈，需要再次 resume 或重新启动下一轮。

解决方法：前端增加了两层控制。第一层是支持 interrupt / resume 机制，自动识别工作流是否要求续跑；第二层是增加兜底逻辑：如果这一轮已经有评分、但没有下一题，并且还没达到总轮次，前端会主动请求下一题。

### 问题 3：工作流字段名不统一，前后端容易接错

现象：同样的工作流逻辑在部分版本能运行，在另一些版本里会出现参数识别不到的问题。

原因：不同节点可能使用 resume_file、resume、file、document 等不同名称。

解决方法：前端采取兼容性字段并行传递的方案，尽可能同时传递文件对象字段和文本字段，提高对不同工作流版本的适配能力。

### 问题 4：直接部署到公网会泄露 Token

现象：原始版本前端直接调用 Coze API，需要把工作流 Token 写在浏览器里。

风险：一旦项目开放到公网，任何用户都可能在浏览器里看到 Token，存在明显安全隐患。

解决方法：增加服务端代理层，把 COZE_WORKFLOW_ID、COZE_WORKFLOW_TOKEN、COZE_ACCESS_TOKEN 放到服务端环境变量中。前端只调用本站的 /api 接口，由服务端再转发到 Coze。这样就实现了“前端可公网访问，但密钥不暴露”。

## 六、当前方案的价值

- 实现了从“能调用模型”到“能支撑真实求职场景”的完整流程。
- 兼顾了简历优化和模拟面试两个功能模块。
- 通过文本提取、多字段兼容、续跑控制，提升了工作流稳定性。
- 通过服务端代理完成了公网部署安全改造。

## 七、适合 PPT 的总结表述

这个项目不是简单地接入一个大模型，而是围绕“简历优化”和“模拟面试”两个真实场景，完成了工作流编排、前端交互、文件处理、多轮会话控制和公网部署安全改造。核心难点主要集中在文档解析稳定性、工作流参数兼容、多轮面试续跑控制，以及部署时的 Token 安全问题。最终，我们通过“前端提取简历正文 + 多字段兼容传参 + interrupt/resume 机制 + 服务端代理部署”的方案，解决了这些关键问题。
